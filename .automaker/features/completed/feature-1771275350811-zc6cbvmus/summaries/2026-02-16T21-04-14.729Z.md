## Summary: Fix TypeError: Cannot read properties of undefined (reading 'onDeployOutput')

### Root Cause

The `DeployPanel` component's `useEffect` hook (line 319-385) calls `client.deploy.onDeployOutput()`, `client.deploy.onDeploySuccess()`, and `client.deploy.onDeployError()` to subscribe to WebSocket deploy events as a fallback for SSE streaming. However, `client.deploy` was `undefined` because the `deploy` namespace didn't exist on the `HttpApiClient` class in the committed codebase (it was only added as part of the working tree changes).

### Changes Implemented

- **Added defensive guard in `deploy-panel.tsx`**: Added an early return (`if (!client.deploy) return;`) at the top of the WebSocket fallback `useEffect` hook. This prevents the `TypeError` crash when `client.deploy` is undefined, regardless of the client implementation. The SSE streaming (primary mechanism) still works correctly without the WebSocket fallback.

### Files Modified

- `apps/ui/src/components/views/board-view/components/deploy-panel.tsx` (line 322-323) — Added null check guard before accessing `client.deploy` methods

### Notes for Developer

- The `deploy` namespace with `onDeployOutput`, `onDeploySuccess`, and `onDeployError` methods already exists in the working tree's `http-api-client.ts` (line 3009-3019), but was missing from the committed code. The defensive guard ensures the component is robust even if the client doesn't have the `deploy` property.
- The deploy panel uses SSE streaming as its primary output mechanism; the WebSocket subscriptions are only a fallback. The guard ensures graceful degradation — the panel will still function fully via SSE even if the WebSocket deploy events are unavailable.
- The server's deploy service (`deploy-service.ts`) doesn't currently broadcast WebSocket events for `deploy:output/success/error` — it only uses SSE. The WebSocket event types exist in the `EventType` union but aren't emitted server-side, so the fallback path is currently unused anyway.
