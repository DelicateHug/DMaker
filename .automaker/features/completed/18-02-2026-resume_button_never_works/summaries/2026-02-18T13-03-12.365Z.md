## Summary: Resume Button Never Works

### Root Cause Analysis

The resume button was failing because `FeatureLoader.get()` could not find the feature on disk in several edge cases:

1. **Cold cache after `migrateToStatusLayout` completed**: The migration runs once per project lifetime and populates the `featureDirCache`. If the initial `resolveFeatureDir` scan failed to find a feature (e.g., due to timing, Windows FS latency, or the feature being in a non-standard directory), the fallback used `getFeatureDir()` which constructs a **legacy month-based path** (e.g., `features/2026-february/featureId/`) instead of the correct status-based path (e.g., `features/in_progress/featureId/`). This caused the feature to appear as "not found."

2. **Silent failure pattern**: The resume endpoint returned `{ success: true }` immediately via fire-and-forget, so when `resumeFeature` failed asynchronously, the UI never learned about the error. The user saw the feature briefly appear to resume, then nothing happened.

3. **Missing error type**: The `feature_not_found` error type emitted by `resumeFeature` was not in the UI's TypeScript type union, potentially causing error handling inconsistencies.

### Changes Implemented

1. **`feature-loader.ts` - `get()` method**: Added a retry mechanism that performs a full `collectFeatureDirs()` scan when `resolveFeatureDir()` returns null. This ensures the directory cache is fully repopulated before falling back to the legacy path, catching features that the initial scan missed.

2. **`feature-loader.ts` - `resolveFeatureDir()` method**: Added a secondary scan pass for non-status, non-month directories (catches features in custom pipeline directories or other non-standard locations).

3. **`resume-feature.ts` route handler**: Added a synchronous `featureLoader.get()` validation call before returning `{ success: true }`. This eagerly populates the cache and returns a clear `{ success: false, error: "..." }` to the client if the feature cannot be found, instead of silently failing in the background.

4. **`use-board-actions.ts` UI handler**: Added toast error notifications when resume fails, so the user gets visible feedback instead of a silent failure.

5. **`electron.d.ts`**: Added `'feature_not_found'` to the `errorType` union for `auto_mode_error` events.

### Files Modified

- `apps/server/src/services/feature-loader.ts` - Added retry with `collectFeatureDirs` in `get()`, added extended scan in `resolveFeatureDir()`
- `apps/server/src/routes/auto-mode/routes/resume-feature.ts` - Added pre-validation of feature existence before fire-and-forget
- `apps/ui/src/components/views/board-view/hooks/use-board-actions.ts` - Added toast notifications on resume failure
- `apps/ui/src/types/electron.d.ts` - Added `feature_not_found` error type

### Notes for Developer

- The 2 test failures in `claude-usage-service.test.ts` are pre-existing timeout issues with `vi.useFakeTimers()`, unrelated to these changes
- All 95 tests in `feature-loader.test.ts` (81) and `auto-mode-service.test.ts` (14) pass
- The fix addresses both the immediate symptom (feature not found) and the UX issue (silent failure with no user feedback)
