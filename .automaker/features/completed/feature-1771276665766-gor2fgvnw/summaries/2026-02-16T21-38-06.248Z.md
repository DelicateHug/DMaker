## Summary: Fix deploy scripts not producing output

### Root Causes Identified

Three issues were preventing deploy script output from being displayed in the UI:

1. **PowerShell `Write-Host` output not captured (Windows)**: When using `-File` parameter with `powershell.exe` (PowerShell 5.1), `Write-Host` output goes directly to the console host, bypassing the stdout pipe. This meant all PowerShell scripts using `Write-Host` produced no visible output.

2. **No WebSocket fallback for deploy events**: The frontend has WebSocket fallback code for when SSE streaming fails, subscribing to `deploy:output`, `deploy:success`, and `deploy:error` events. However, the server **never emitted these events** via the WebSocket event emitter - it only sent data through SSE. If SSE streaming failed for any reason, the UI would show no output.

3. **SSE connection reliability**: The SSE stream had no initial handshake data, so the client couldn't confirm the stream was active. Added an initial `:ok` SSE comment to force any buffering middleware/proxies to flush and confirm the connection.

### Changes Implemented

**Server - `apps/server/src/services/deploy-service.ts`**:

- Changed PowerShell `.ps1` execution from `-File` to `-Command` with `*>&1` stream redirection, which merges all PowerShell output streams (including `Write-Host`'s information stream) into stdout for proper pipe capture
- Added `windowsHide: true` to prevent visible console windows for spawned processes on Windows
- Added `PYTHONIOENCODING: 'utf-8'` env var for Python scripts on Windows

**Server - `apps/server/src/routes/deploy/routes/run.ts`**:

- Added WebSocket event emission alongside SSE streaming: `deploy:output` for stdout/stderr, `deploy:success`/`deploy:error` for completion
- Added initial `:ok\n\n` SSE comment to confirm connection and flush buffering middleware
- Added `req.socket.setTimeout(0)` to prevent socket timeouts during long-running scripts
- Accepts optional `EventEmitter` parameter for WebSocket broadcast

**Server - `apps/server/src/routes/deploy/index.ts`**:

- Updated `createDeployRoutes` to accept and pass through an `EventEmitter` parameter

**Server - `apps/server/src/index.ts`**:

- Pass `events` emitter to `createDeployRoutes(deployScriptRunner, events)`
- Made compression filter more robust with `startsWith` check for `text/event-stream`

**Frontend - `apps/ui/src/components/views/board-view/components/deploy-panel.tsx`**:

- Set `sseReceivedDataRef.current = true` as soon as the reader receives **any** data (including SSE comments), not just parsed events. This prevents the 10-second timeout from aborting the SSE connection and falling through to the (previously broken) WebSocket fallback.

### Files Modified

- `apps/server/src/services/deploy-service.ts` - PowerShell spawn fix with `*>&1` stream redirect
- `apps/server/src/routes/deploy/routes/run.ts` - WebSocket event emission + SSE connection handshake
- `apps/server/src/routes/deploy/index.ts` - Accept EventEmitter parameter
- `apps/server/src/index.ts` - Pass events to deploy routes, robust compression filter
- `apps/ui/src/components/views/board-view/components/deploy-panel.tsx` - Early SSE connection detection

### Notes for Developer

- All 1278 server unit tests pass with no regressions
- The WebSocket fallback is now functional, providing a reliable secondary channel when SSE streaming encounters issues (proxies, Electron environments, etc.)
- The PowerShell fix uses `-Command` with stream redirection (`*>&1`) instead of `-File`. This correctly captures `Write-Host` output in PowerShell 5.1 (Windows default). Single quotes in script paths are escaped via `''` (PowerShell's escape for single quotes within single-quoted strings).
- On non-Windows platforms, PowerShell scripts still use `pwsh -File` (PowerShell 7, where `Write-Host` is properly captured)
