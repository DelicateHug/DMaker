## Summary: Fix Claude usage not visible in header

### Problem

The Claude usage popover (`UsagePopover`) was only rendered when the user was on the `/board` route AND when board controls were mounted (`isOnBoardView && boardControls && boardControls.isMounted`). This meant the usage indicator was invisible on all other routes and could also be missing on the board view if the board controls store hadn't fully initialized.

### Root Cause

In `top-nav-bar.tsx` at line 612-616, the `UsagePopover` was gated behind board-view-specific conditions:

```tsx
{
  isOnBoardView &&
    boardControls &&
    boardControls.isMounted &&
    !isTablet &&
    (showClaudeUsage || showCodexUsage) && <UsagePopover />;
}
```

The `BoardHeader` component (which previously rendered `UsagePopover` independently) was commented out and no longer used (`board-view.tsx` line 72-73), so the only remaining usage popover in the header was this one — restricted to the board view.

### Fix

Removed the `isOnBoardView`, `boardControls`, and `boardControls.isMounted` conditions so the usage popover appears globally whenever the user is authenticated with Claude or Codex:

```tsx
{
  !isTablet && (showClaudeUsage || showCodexUsage) && <UsagePopover />;
}
```

This makes the behavior consistent with other global header elements like Settings and Running Agents indicators, which are always visible regardless of which route is active.

### Files Modified

- `apps/ui/src/components/layout/top-nav-bar.tsx` — Removed board-view-only restriction on `UsagePopover` rendering (line 611-612)

### Notes for Developer

- The `!isTablet` check is retained so tablet/mobile users still access usage via the `HeaderMobileMenu` / `MobileUsageBar`
- Pre-existing type errors in the codebase (unrelated to this change) were observed during type-checking
- The `UsagePopover` component is self-contained — it fetches its own data, manages its own state, and reads auth status from `useSetupStore`, so it works correctly outside the board view context
